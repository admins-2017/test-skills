# 权限管理系统需求与架构设计文档

## 1. 项目概述

本项目建设一个企业级后台权限管理系统，用于统一管理后台用户、角色、菜单、按钮、接口和数据访问范围。系统采用前后端分离架构，后端基于 `JDK 17 + Spring Boot 3.x + Maven`，前端基于 `Vue 3 + TypeScript + Element Plus`，数据库使用 `MySQL`，缓存与会话辅助能力使用 `Redis`。

系统第一阶段定位为单体后台管理系统，重点实现稳定、清晰、可扩展的 RBAC 权限模型，并增强按钮级权限控制和数据权限控制能力。

权限控制原则：

- 前端负责页面体验控制，例如菜单是否显示、按钮是否展示、组件是否渲染。
- 后端负责最终安全校验，例如接口是否允许访问、数据是否允许查询或修改。
- 所有关键权限必须在后端二次校验，不能只依赖前端隐藏按钮。
- 菜单权限、按钮权限、接口权限应使用统一权限编码体系。

## 2. 建设目标

### 2.1 业务目标

- 建立统一的后台用户、角色、菜单、按钮、接口和数据权限管理能力。
- 支持管理员灵活配置不同角色可访问的菜单、页面、按钮和数据范围。
- 支持用户登录、退出、刷新令牌、强制下线和在线用户管理。
- 支持权限变更后快速生效，避免用户继续使用旧权限。
- 支持日志审计，记录登录、操作、授权和异常行为。
- 为后续扩展多租户、单点登录、权限中心和字段级权限预留架构空间。

### 2.2 技术目标

- 后端基于 Spring Boot 3.x，适配 JDK 17。
- 前端基于 Vue 3 + TypeScript，提升类型安全和工程可维护性。
- 权限编码前后端统一，降低维护成本。
- 使用 Redis 缓存登录状态、验证码、在线用户和权限数据。
- 使用 MySQL 持久化用户、角色、权限、菜单、部门和日志等数据。
- 后端通过 Spring Security 和自定义权限服务完成接口鉴权。
- 数据权限在后端统一兜底，避免前端绕过。

### 3.3 请求调用链路

典型接口请求流程：

1. 前端 Axios 请求携带 `Authorization: Bearer accessToken`。
2. 后端 `JwtAuthenticationFilter` 解析 Token。
3. 从 Redis 或数据库加载当前登录用户信息。
4. Spring Security 建立认证上下文。
5. Controller 进入具体接口。
6. `@PreAuthorize` 调用权限服务判断按钮或接口权限。
7. 如接口涉及数据查询，数据权限切面追加数据范围条件。
8. Service 执行业务逻辑。
9. Mapper 查询或写入 MySQL。
10. 操作日志切面记录请求、响应和异常信息。
11. 后端返回统一响应结构。

## 5. 核心业务需求

### 5.1 登录认证

#### 功能说明

用户通过账号、密码和验证码登录系统。登录成功后，后端返回访问令牌和刷新令牌。前端保存令牌，并在后续请求中通过请求头携带访问令牌。

#### 业务规则

- 用户账号不存在时，统一提示账号或密码错误。
- 密码错误时，记录失败次数。
- 用户被禁用时，不允许登录。
- 登录成功后写入登录日志。
- 登录成功后返回 Token，前端再调用当前用户信息接口。
- 支持退出登录，退出后清理 Redis 中的在线状态。
- 支持 Token 过期刷新。
- 支持管理员强制用户下线。

#### 登录接口

```text
POST /auth/login
```

请求参数：

```json
{
  "username": "admin",
  "password": "123456",
  "captchaKey": "captcha-key",
  "captchaCode": "abcd"
}
```

响应数据：

```json
{
  "accessToken": "xxx",
  "refreshToken": "xxx",
  "expiresIn": 7200
}
```

### 5.6 按钮级权限控制

#### 功能目标

按钮级权限控制用于控制用户在页面内可以执行哪些操作，例如新增、编辑、删除、导出、审核、分配角色、重置密码等。

#### 设计原则

- 按钮权限必须作为菜单树中的 `BUTTON` 类型节点进行维护。
- 每个按钮必须有唯一权限编码。
- 前端根据权限编码决定按钮是否展示。
- 后端根据同一权限编码决定接口是否允许调用。
- 权限编码前后端统一维护，不能出现前端一套、后端一套。

#### 权限编码规范

格式：

```text
模块:资源:动作
```

示例：

```text
system:user:add
system:user:edit
system:user:delete
system:role:assign-permission
system:menu:add
system:dept:delete
```

#### 前端实现方式

方式一：权限指令。

```vue
<el-button
  v-permission="'system:user:add'"
  type="primary"
>
  新增用户
</el-button>
```

方式二：权限组件。

```vue
<PermissionGate code="system:user:delete">
  <el-button type="danger">删除</el-button>
</PermissionGate>
```

方式三：权限方法。

```ts
if (hasPermission('system:user:edit')) {
  openEditDialog(row)
}
```

方式四：封装权限按钮。

```vue
<PermissionButton
  code="system:user:add"
  type="primary"
  @click="openCreateDialog"
>
  新增
</PermissionButton>
```

#### 后端实现方式

接口方法使用权限注解：

```java
@PreAuthorize("@perm.has('system:user:add')")
@PostMapping("/users")
public Result<Long> createUser(@RequestBody UserCreateReqVO reqVO) {
    return Result.success(userService.createUser(reqVO));
}
```

权限判断逻辑：

- 获取当前登录用户。
- 如果是超级管理员，直接放行。
- 从 Redis 或数据库获取用户权限集合。
- 判断权限集合中是否包含当前权限编码。
- 不存在则返回 `403 Forbidden`。

### 5.7 接口权限控制

#### 功能说明

接口权限用于防止用户绕过前端页面直接调用后端接口。

#### 控制方式

- 对核心接口添加 `@PreAuthorize`。
- 后端统一维护权限编码。
- 可选增加接口资源表，用于记录接口路径与权限编码关系。
- 对开放接口，如登录、验证码、刷新令牌，不做登录拦截。
- 对已登录接口，必须通过 JWT 鉴权。
- 对敏感接口，必须额外检查按钮或操作权限。

#### 接口权限示例

```text
POST   /system/users               system:user:add
PUT    /system/users/{id}          system:user:edit
DELETE /system/users/{id}          system:user:delete
PUT    /system/users/{id}/roles    system:user:assign-role
PUT    /system/users/{id}/password system:user:reset-password
```

### 5.8 数据权限控制

#### 功能说明

数据权限用于控制用户能查看或操作哪些数据。常见于用户列表、日志列表、业务单据列表等场景。

#### 数据权限类型

```text
ALL              全部数据
DEPT_ONLY        本部门数据
DEPT_AND_CHILD   本部门及下级部门数据
SELF             仅本人数据
CUSTOM_DEPT      自定义部门数据
```

#### 配置位置

数据权限配置在角色上。一个用户拥有多个角色时，数据权限取最大范围或合并范围。

推荐策略：

- 如果任一角色拥有全部数据，则拥有全部数据。
- 如果多个角色拥有自定义部门，则合并部门范围。
- 如果角色同时拥有本部门和自定义部门，则取并集。
- 仅本人权限只在没有更大范围时生效。

#### 后端实现方式

定义数据权限注解：

```java
@DataScope(deptAlias = "d", userAlias = "u")
@GetMapping("/users")
public Result<PageResult<UserRespVO>> pageUsers(UserPageReqVO reqVO) {
    return Result.success(userService.pageUsers(reqVO));
}
```

数据权限拦截器根据当前用户的数据范围，动态追加查询条件。

示例 SQL 条件：

```sql
AND u.dept_id IN (...)
```

或：

```sql
AND u.create_user_id = 当前用户ID
```

#### 数据权限适用场景

- 用户管理列表
- 部门人员列表
- 操作日志列表
- 登录日志列表
- 业务数据列表
- 导出数据

### 5.9 部门管理

#### 功能说明

部门管理用于维护组织架构，是用户归属和数据权限过滤的重要基础数据。

#### 功能清单

- 部门树查询
- 新增部门
- 编辑部门
- 删除部门
- 启用 / 禁用部门
- 调整部门排序
- 配置部门负责人

#### 业务规则

- 部门采用树形结构。
- 禁用部门后，该部门下用户仍保留部门关系，但新增用户不可选择禁用部门。
- 删除部门前需要校验是否存在子部门或关联用户。
- 部门用于数据权限范围计算。

### 7.5 权限缓存设计

Redis Key 设计：

```text

login:refresh:{refreshToken}
```

缓存策略：

- 用户登录时加载权限并缓存。
- 每次请求优先从 Redis 获取用户登录态。
- 权限变更后删除受影响用户的 `permission:user:{userId}`。

- 用户禁用后强制清除登录态。
- 角色授权变更后，清理该角色下所有用户的权限缓存。

## 8. 前端核心设计

### 8.1 登录流程

前端登录流程：

1. 用户输入账号、密码、验证码。
2. 调用 `POST /auth/login`。
3. 保存 `accessToken` 和 `refreshToken`。
4. 调用 `GET /auth/me`。
5. 保存用户信息、角色、菜单、按钮权限。
6. 根据菜单动态生成路由。
7. 跳转首页。

## 11. 安全设计

### 11.1 密码安全

- 密码使用 BCrypt 加密存储。
- 不允许明文保存密码。
- 重置密码后建议用户首次登录修改密码。
- 登录失败次数过多时，可临时锁定账号。

### 11.2 Token 安全

- Access Token 有较短有效期。
- Refresh Token 有较长有效期。
- Refresh Token 保存在 Redis 中。
- 用户退出后清理 Token。
- 用户禁用后强制下线。

### 11.3 接口安全

- 所有管理接口默认需要登录。
- 关键接口需要权限编码校验。
- 删除、授权、重置密码等敏感操作记录审计日志。
- 请求参数进行后端校验。
- 防止越权访问其他部门或其他用户数据。

### 11.4 前端安全

- 不在前端保存明文密码。
- Token 建议保存在安全存储策略中，并避免暴露给第三方脚本。
- 路由守卫中校验登录状态。
- 按钮隐藏仅作为用户体验，不作为安全边界。
- 前端刷新页面后，应重新加载当前用户权限。

## 15. 设计结论

本系统采用 RBAC 作为核心权限模型，并扩展按钮级权限和数据权限。整体设计兼顾易实现、易维护和后续扩展能力。

第一版建议重点保证：

- 权限编码统一。
- 菜单、按钮、接口三端一致。
- 后端接口必须鉴权。
- 数据权限必须后端兜底。
- 权限变更后缓存及时失效。
- 前端权限组件和指令统一封装。

该设计可以支撑大多数企业后台系统的权限管理需求，并为后续演进为权限中心、多租户平台或统一 IAM 系统保留扩展空间。
