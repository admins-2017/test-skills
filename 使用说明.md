# Agent Harness 使用说明

本文说明如何使用 `agent-harness-generator` 生成的内容推进权限管理系统开发。当前 harness 的核心文件包括：

- `AGENTS.md`：智能体开发规范
- `architecture.md`：从需求文档提取的架构摘要
- `task.json`：任务 DAG、依赖关系、验收标准和验证命令
- `env.json`：工具、服务、MySQL、Redis、端口等环境快照
- `init.ps1`：环境初始化脚本
- `run-loop.ps1`：自动调度任务、调用 Codex、验证、提交的循环脚本

## 方向一：在 Codex 中按计划让智能体开发

这种方式适合你希望逐个任务审查、人工把控节奏，或者先用 Codex 对某个任务做精细开发。

### 1. 先让 Codex 读取上下文

在 Codex 中打开当前项目目录后，可以这样发起：

```text
请先阅读 AGENTS.md、architecture.md、权限管理系统需求设计文档.md 和 task.json。
根据 task.json 的依赖关系，找出下一个可以执行的 pending 任务，只实现这个任务。
实现后运行该任务的 e2e_test 验证命令，并汇报修改文件和验证结果。
不要修改 task.json 的 status 字段。
```

如果你想指定任务，例如第一个后端任务：

```text
请执行 task.json 中的 W01-J01。
先阅读 AGENTS.md 和 architecture.md，严格按任务 description 和 acceptance 实现。
完成后运行 e2e_test: mvn -f permission-system-backend/pom.xml -DskipTests compile。
不要改其他任务，不要修改 task.json 的 status 字段。
```

### 2. Codex 开发时应该遵守的规则

让智能体始终以 `task.json` 为任务边界：

- 一次只做一个任务，避免跨任务大改。
- 先看 `dependencies`，依赖未完成的任务不要抢跑。
- 代码风格和安全要求以 `AGENTS.md` 为准。
- API、表结构、权限编码以 `architecture.md` 和原始需求文档为准。
- 完成后必须跑当前任务的 `e2e_test`。
- 如果任务包含 `e2e_services`、`e2e_api_test`、`e2e_playwright`，需要启动服务后做对应验证。

### 3. 推荐的人工推进流程

1. 运行一次任务预览：

```powershell
powershell -ExecutionPolicy Bypass -File .\run-loop.ps1 -DryRun
```

2. 选取最前面的 pending 任务。
3. 在 Codex 中指定该任务 ID，让它实现代码。
4. 查看 Codex 汇报的修改文件和验证结果。
5. 你确认没问题后提交代码，或让 Codex 帮你提交。
6. 继续下一个任务。

### 4. 常用提示词模板

```text
请执行 task.json 中的 {任务ID}。

要求：
1. 阅读 AGENTS.md、architecture.md、权限管理系统需求设计文档.md。
2. 只实现 {任务ID}，不要顺手做后续任务。
3. 严格满足 task.json 中该任务的 description 和 acceptance。
4. 完成后运行 e2e_test。
5. 如果验证失败，先修复再汇报。
6. 不要修改 task.json 的 status 字段。

最后请汇报：
- 修改了哪些文件
- 验证命令和结果
- 是否还有风险或需要我确认的点
```

## 方向二：在 PowerShell 中直接用 Codex 执行任务

这种方式适合你希望让脚本自动读取任务、调用 Codex、运行验证、失败重试并持续推进。

### 1. 初始化环境

首次使用前运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\init.ps1
```

如果只想先初始化文件结构，不检测 MySQL/Redis：

```powershell
powershell -ExecutionPolicy Bypass -File .\init.ps1 -SkipInfra
```

当前配置的基础设施是：

- MySQL：`8.146.233.212:3306`
- MySQL 用户名/密码：`ytzy / <MYSQL_PASSWORD>`，提交到 GitHub 的配置中密码使用 `CHANGE_ME` 占位，请在本地运行前替换为真实密码。
- Redis：`8.146.233.212:6379`
- Redis database：`3`

### 2. 预览任务执行顺序

```powershell
powershell -ExecutionPolicy Bypass -File .\run-loop.ps1 -DryRun
```

这个命令不会调用 Codex，也不会改代码，只会按 DAG 展示可执行顺序。

### 3. 自动执行全部任务

```powershell
powershell -ExecutionPolicy Bypass -File .\run-loop.ps1
```

脚本会自动做这些事：

- 从 `task.json` 找到依赖已完成的 pending 任务。
- 构造任务 prompt。
- 调用 `codex.cmd` 执行开发。
- 运行任务的 `e2e_test`。
- 按需启动 `e2e_services`。
- 执行 API 或 Playwright 测试。
- 验证失败时自动重试。
- 验证通过后提交代码。
- 写入 `progress.log` 和 `logs/`。

### 4. 只执行少量任务

只执行 1 个任务：

```powershell
powershell -ExecutionPolicy Bypass -File .\run-loop.ps1 -MaxTasks 1
```

只执行 3 个任务：

```powershell
powershell -ExecutionPolicy Bypass -File .\run-loop.ps1 -MaxTasks 3
```

### 5. 从指定任务恢复

如果某个任务失败，例如 `W03-J04`，修复环境或代码后可以从该任务继续：

```powershell
powershell -ExecutionPolicy Bypass -File .\run-loop.ps1 -StartFrom W03-J04
```

### 6. 指定 Codex 模型和推理等级

```powershell
powershell -ExecutionPolicy Bypass -File .\run-loop.ps1 -Model gpt-5.4 -Level xhigh
```

### 7. 运行时产物

- `task.json`：任务状态会由脚本更新。
- `progress.log`：记录任务开始、完成、失败、重试。
- `logs/codex-{taskId}-attempt1.log`：每次 Codex 调用日志。
- `logs/{serviceName}.log`：服务启动日志。
- `.tmp-codex-prompt.md`：临时 prompt 文件。

### 8. 常见注意事项

- PowerShell 必须使用 `-ExecutionPolicy Bypass -File` 执行脚本。
- `.ps1` 文件必须保持 UTF-8 with BOM。
- `task.json` 和 `env.json` 必须保持 UTF-8 BOM-free。
- MySQL 和 Redis 只做 TCP 连通性探测，不验证账号密码是否能登录。
- 如果任务需要启动后端，确保 MySQL 和 Redis 真实可用。
- 如果前端 E2E 失败，优先查看 `permission-system-web/tests/e2e/results` 和 `logs/permission-system-web.log`。

## 推荐使用方式

早期建议先用 Codex 手动跑前 2 到 3 个任务，确认项目骨架风格符合预期。等后端骨架、前端骨架和基础配置稳定后，再用 PowerShell 的 `run-loop.ps1 -MaxTasks N` 小批量自动推进。
